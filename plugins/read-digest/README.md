# read-digest

> 臃肿的稿子先抽骨架，再读 / 再改。

一个**先抽结构、再分叉**的 skill：别人或 AI 丢来一份读着很累的臃肿稿，`read-digest` 先把整篇倒着抽成「一句话一段」的反向大纲骨架，让它的真实结构和结构病（重复、跑题、断层、头重脚轻）一眼现形；然后照你的需要——只读懂就停、要改就逐段下具体指令、要交付就照骨架重写出一份干净精简稿。

引擎是写作里的 **reverse outline（反向大纲）**：写完后倒着抽大纲、检查结构的老技法，这里挪用来读懂 + 改顺别人/AI 给的稿。

## 为什么需要它

AI 生成（或别人写得啰嗦）的文档读着累，病几乎都在**结构层**，不在字面：

- 同一个点换三种说法重复说（"提效"能讲满三段）
- 套话填充密度高（"数字化转型的时代背景下""毋庸置疑""不仅如此"）
- 结论 / 唯一有信息量的那句埋在最后（头重脚轻）
- bullet-soup：本该一句话的东西摊成五个 bullet

所以逐字顺着读最吃力，光把每句改漂亮也没用。先看清骨架，才读得快、改得准、知道砍哪。

## 触发方式

直接说出以下任意一种（中英文都行）：

- "读 AI 写的文档很累 / 很难受"
- "这份文档太啰嗦 / 太臃肿，帮我看清结构"
- "这稿子重复太多，哪些能砍"
- "边读边改这份 AI 稿"
- "把这份长文 / PRD / 报告抽个结构 / 反向大纲出来"
- "这份报告读着绕，帮我理清主干"
- "先帮我看懂这份文档到底说了啥再改"

或者直接 `/read-digest`。

## 工作流程

```
1  先通读一遍，不动手
2  抽反向大纲骨架：每段一句话说它"实际在干嘛"（不是抄标题）
3  在骨架上标结构病：重复 / 跑题 / 断层 / 错位 / 头重脚轻
4  停下来等你选：
   (a) 只想读懂 → 骨架就够了，指出哪几段纯水可跳过，到此为止
   (b) 想改     → 逐段下 craft 级指令（具体说哪儿不对），一段段 accept/reject
   (c) 要精简稿 → 照骨架重写干净版，另存 xxx-digest.md，原文不覆盖
5  (可选) 复杂结构转 mermaid/流程图辅助，但以骨架为准
```

## 易翻车的地方

- **别把"啰嗦"和"有信息"混为一谈**：有的段落啰嗦，但裹着 load-bearing 的具体信息（数字、约束、决策）。压缩时留信息、删套话，拿不准就标出来问。
- **骨架抽"实际在干嘛"不是抄标题**：AI 稿常挂羊头卖狗肉，照标题抄就把毛病抄进骨架。
- **改的时候别只顺句子、不动结构**：病在结构层，每句改漂亮了整体还是绕。
- **出精简稿不覆盖原文**：原文可能还要回去核对。

## 边界（重要区分）

- **不要和"总结 / summary"混**：summary 替你读完、给浓缩版，你不再碰原文；reverse outline 给你原文的结构图，你还要回原文去读和改。目的是**辅助读和改，不是代替读**。
- **跟 `humanizer` 的分工**：`read-digest` 是 **input 侧**——读懂 / 改顺别人或 AI 丢来的臃肿稿；`humanizer` 是 **output 侧**——把自己手上、尤其 AI 写的稿清成人能读懂的成品（去 AI 味 / 去翻译腔 / 换白话 / 清过程痕迹，原 fair-copy 已并入它）。方向相反，互补成对。

## 一句话总结

**先把臃肿稿抽成"一句话一段"的骨架，让结构病现形；再照需要——读懂就停、要改下具体指令、要交付照骨架重写。骨架抽"实际在干嘛"不是抄标题，压缩时留干货删套话，原文务必留底。**
