# Read-Digest — 把臃肿的稿子先抽骨架，再读 / 再改

## 这个 skill 在解决什么

AI 生成（或别人写得啰嗦）的文档读着累，原因几乎都在**结构层**，不在字面：

- 同一个点换三种说法重复说（"提效"能讲满三段）。
- 套话填充密度高（"数字化转型的时代背景下""毋庸置疑""值得一提""不仅如此"）。
- 结论 / 唯一有信息量的那句埋在最后（头重脚轻）。
- bullet-soup：本该一句话的东西摊成五个 bullet。

所以**逐字顺着读最吃力**——你得透过一层层包裹去猜骨架。光把每句改漂亮也没用，因为病在结构。

这个 skill 的引擎是 **reverse outline（反向大纲）**：先把整篇倒着抽成"一句话一段"的骨架，让真实结构（和它的毛病）一眼可见。看清骨架之后，读得快、改得准，要砍也知道砍哪。

**类比锚点**（帮快速理解在干嘛）：
- 写作 / 学术：reverse outline —— 写完后倒抽大纲，检查结构、找重复与断层。
- 读陌生代码：先看函数签名和调用图，不一上来逐行啃实现。
- 地铁：先看线路图（骨架）决定在哪站下，而不是沿街一步步走完。

## 工作流

第一步永远是抽骨架；抽完才分叉。

1. **先通读一遍，不动手。** 整篇过一遍，先不改、不删、不评论。
2. **抽反向大纲骨架。** 给每一段（或每一小节）写**一句话**，说它**"实际在干嘛"**——它真正的功能 / 主张，**不是它的标题或它自称在干嘛**。串成骨架贴回对话里给用户看。
   > ⚠️ 抽的是"实际在干嘛"。AI 稿常挂羊头卖狗肉（标题写 A、内容跑到 B），照标题抄你就把这个毛病抄进骨架了。
3. **在骨架上标结构病。** 逐项扫：
   - **重复**：多段在说同一个点 → 标出来，能缩成一句。
   - **跑题**：某段跟主线无关。
   - **断层**：逻辑跳跃、缺论据、前后接不上。
   - **错位**：一段里塞了好几个主题（读者跟不上）。
   - **头重脚轻**：唯一有信息量的内容埋在最后 / 结论没提前。
4. **分叉——问用户要哪条路（默认先问，别自作主张往下改）：**
   - **(a) 只想读懂** → 骨架就够了。把主干讲清，并指出哪几段是纯水、可以直接跳过。到此为止。
   - **(b) 想改** → 逐段沟通，但下的是 **craft 级指令**（具体说哪儿不对），不是 vibes。一段一段 accept / reject，别一次性大改。
     > 反例（vibes）："把这段改好" / "这段读着难受" —— 等于没说，AI 会再给你一版同样空的。
     > 正例（craft）："这三段在重复'提效'，留最狠的一句、其余删" / "这里缺论据，补一个具体数字或例子" / "口径不一，统一成 X"。
   - **(c) 想要精简稿** → 照骨架重写出一份干净可读版：砍重复、砍套话、bullet-soup 收成 tight 结构或散文、结论提前。**另存新文件（如 `xxx-digest.md`），不覆盖原文。**
5. **（可选）画图辅助。** 文档结构复杂、或本身在讲流程 / 依赖 / 时序时，把骨架转成 mermaid / 流程图 / 思维导图，看全局、抓断层。
   > ⚠️ 图是"快速进入 + 发现断层"用的，**不替代骨架**。AI 自动生成的图可能有逻辑遗漏，一律拿回第 2 步那张骨架核对，以骨架为准。

## ⚠️ 易翻车的地方

- **别把"啰嗦"和"有信息"混为一谈。** 有的段落写得啰嗦，但裹着 load-bearing 的具体信息（数字、约束、决策、谁要做什么）。压缩时**留信息、删包裹的套话**，别把内核一起扔了。拿不准就标出来问，别静默删。
- **骨架抽"实际在干嘛"，不是抄标题**（见上）。
- **改的时候别只顺句子、不动结构**——AI 稿的病在结构层，每句都改漂亮了，整体还是绕。
- **出精简稿不覆盖原文**——原文可能还要回去核对，或对方还要用。
- **这不是替用户读**。骨架是帮他更快回到原文，不是给个摘要让他从此不看原文。

## 核心提示词模板

skill 的核心动作，也可直接复制给任何 AI 用。

**完整版（读 + 改一份 AI 长稿）：**

> 这份文档读着很累。先别改，按下面来：
> 1. 通读一遍，给每一段写一句话，说它**实际在讲什么**（它真正的功能 / 主张，不是它的标题）。串成一张反向大纲骨架给我。
> 2. 在骨架上标出结构问题：哪几段在**重复**、哪段**跑题**、哪里逻辑**断层**、哪段塞了**多个主题**、有没有**头重脚轻**（结论 / 干货埋太后）。
> 3. 标完**停下来等我选**：我要的是「只读懂」「逐段改」还是「出一份精简稿」。
> 4. 如果是逐段改，我会一段一段给你具体指令，你别自作主张大改；如果某段我拿不准是水还是干货，先标出来问我，别直接删。
> 5. 如果出精简稿，照骨架重写干净可读版（砍重复和套话、结论提前），**另存为 `xxx-digest.md`，原文别动。**

**短版（随手理一份）：**

> 把这份文档先抽成反向大纲——每段一句话说它实际在讲什么，然后标出哪几段在重复、哪里逻辑断了、哪段最该先读、哪段是纯水可跳过。先别改，等我说要不要改。

## 例子

**Input（AI 生成的臃肿片段）：**

```markdown
## 二、AI 赋能投研的价值与意义

在当今数字化转型的时代背景下，人工智能技术正在深刻地改变着投资研究的方方面面。
AI 能够极大地提升投研效率，这一点是毋庸置疑的。

具体来说，AI 可以帮助投研人员更快速、更高效地处理海量信息。通过自动化的方式，
AI 能够显著减少人工处理信息所需的时间成本。

此外，值得一提的是，AI 在信息处理方面的能力也是非常突出的，它可以高效完成大量
重复性的信息处理工作，这无疑会带来效率上的巨大提升。

不仅如此，AI 还能辅助决策。基于大模型的解析能力，系统可对企业财报、BP 等材料
提取关键信息——例如对一家拟投企业的三年财报，可在数分钟内输出营收结构、毛利率
趋势与主要风险点三项结构化结论。
```

**Output（反向大纲骨架 + 诊断）：**

```
骨架（每段实际在干嘛）：
- 段1：口号——"AI 在改变投研、能提效"（无具体信息）
- 段2：AI 更快处理海量信息、省人工时间（= 提效，同段1）
- 段3：AI 擅长重复性信息处理、带来效率提升（= 又说了一遍提效）
- 段4：AI 解析财报/BP 提关键信息 → 例：三年财报数分钟出营收结构/毛利率趋势/
       风险点三项结论（← 全节唯一 load-bearing 的具体信息）

诊断：
- 重复：段1/2/3 是同一件事（提效），三段可缩成一句。
- 头重脚轻：唯一有信息量的是段4那个具体例子，却埋在最后。
- 套话密度高："数字化转型时代背景""毋庸置疑""值得一提""不仅如此"全是填充，可删。

→ 若出精简稿，这一节其实就一句：
"AI 可解析财报、BP 等非结构化材料并输出结构化结论——如三年财报数分钟内得出
营收结构、毛利率趋势、主要风险点——把投研人力从信息搬运中释放出来。"
```

（保留了段4的具体信息这个内核；砍掉的是包裹它的套话和三段重复的"提效"口号。注意：是先看清骨架才敢这么砍，而不是逐句硬压。）

## 命名与边界（备忘）

- **引擎是 reverse outline（反向大纲）**：写完后倒着抽大纲、检查结构的老技法。这里把它从"检查自己的稿"挪用到"读懂 + 改顺别人/AI 给的稿"。
- **跟 `humanizer` 的分工**：`humanizer` 是 **output 侧**——把自己手上、尤其 AI 写的稿，清成人能读懂的成品（去 AI 味 / 去翻译腔 / 换白话 / 清过程痕迹，原 fair-copy 已并入它）。`read-digest` 是 **input 侧**——把别人/AI 丢来的臃肿稿读懂、改顺。一个处理"我要交出去的稿"，一个处理"别人给我的臃肿"，方向相反。
- **不要和"总结/summary"混**：summary 替你读完、给浓缩版，你不再碰原文；reverse outline 给你原文的结构图，你还要回原文去读和改。目的是**辅助读和改，不是代替读**。

## 一句话总结

**先把臃肿稿抽成"一句话一段"的骨架，让结构病现形；再照需要——读懂就停、要改下具体指令、要交付照骨架重写。骨架抽"实际在干嘛"不是抄标题，压缩时留干货删套话，原文务必留底。**
