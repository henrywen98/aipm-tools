# Fair Copy — 工作稿清成交付稿

## 这个 skill 在解决什么

一份反复迭代的工作稿里，其实混着两类信息：

- **给自己看的过程信息**：修订历史、改动说明、待确认批注、备选写法、给自己的 TODO。
- **读者要的成品内容**：文档本身真正要传达的东西。

交付对象只需要第二类。但作者自己往往看不见第一类的存在——这就是写作里的 **curse of knowledge（知识的诅咒）**：你没法 un-know 自己脑子里的上下文，于是不自觉地留下只有自己懂的指向（"如前所述的改动""上一版里"），读者一头雾水。

这个 skill 做的事，就是按「读者第一次冷读」的视角，把过程痕迹全部剥掉，誊出一份干净成品。

**类比锚点**（帮快速理解在干嘛）：
- 手稿学：foul papers（草稿）→ **fair copy（誊清稿）**，定稿后誊清供正式提交。
- 编辑出版：working copy → **clean copy**，编辑常同时给"带修订追踪版"和"干净版"，但建议从干净版工作。
- 编剧：starred draft（带 `*` 改动标记，给追改动的人）→ **clean draft**（去掉标记，给冷读的人）。
- 开发：一堆 `wip / fix typo / 再改一版` 的乱 commit → **git squash** 成一个连贯的干净 commit。

## 要剥掉的"过程痕迹"清单

逐条对照，命中就清：

1. 修订历史 / changelog / "更新记录" / "版本演进"段落。
2. 版本号、版本说明（`v0.3`、"本版变更点"、"相比 v0.2 的改进"）。
3. 行内批注：`(新增)``(已修改)``(待确认)``(TODO)``(待补)``(?)``【待核】` 这类。
4. 给自己看的旁批 / 备注 / 提醒（"记得问 XX""这里数据待核""先这么写"）。
5. 过程性指向措辞："如前所述的改动""上一版里""之前那版""这版相比上一版…""我后来把它改成了…"。
6. 多处并存的备选写法 / "方案 A or 方案 B" / "这句还是那句？"——**每处只保留你判断为最终的那一个**，其余删掉。
7. 注释掉但没删的段落、`<!-- ... -->` HTML 注释、删除线残留（`~~旧文~~`）、高亮标记。

## ⚠️ 不能当痕迹删掉的（最容易翻车的地方）

有些内容长得像痕迹，**其实是正文**。误删它们比留下痕迹更严重，因为是实质内容丢失。

- 真有一章叫"变更管理""修订记录""发版说明"，**且它本身就是交付内容的一部分**——比如合同里的修订条款、产品文档里读者要看的 changelog、报告里要求保留的版本沿革。保留。
- 业务上本来就在讨论"历史 / 演进 / 迭代"的正文（如"该公司近三年战略演进"）。保留。
- **判断标准**：这一段是"作者对自己改稿过程的元说明"，还是"读者需要的、关于主题本身的内容"？前者删，后者留。
- **拿不准就别静默删**：列出来问用户，宁可多问一句，不要误伤正文。

## 工作流

1. **先判断篇幅与重要性。**
   - 短、低风险的文档：直接清。
   - 长 / 重要 / 正式交付的文档：**先列出"我识别到的所有过程痕迹清单"，请用户确认后再动手**。这一步专门用来挡住误删正文的风险。
2. **清理时只去过程痕迹，严格保留正文实质内容。**
3. **按"读者第一次阅读"的视角重新衔接全文。** 删掉某段后，不能留下指向它的孤儿句（"如上所述"删了上下文后指向空气）。该补的衔接补上。
4. **不要覆盖工作稿。** 输出成新文件（如 `xxx-final.md`），原始工作稿（foul papers）留着——你的思考过程都在里面，删了可惜，下次回溯还要用。
5. **输出干净的完整文档，不要在开头 / 结尾解释"我删了啥、改了啥"**——那本身又是一种过程痕迹，交付对象不需要。

## 核心提示词模板

这是 skill 的核心动作，也可以直接复制给任何 AI 用。

**完整版（长 / 重要文档）：**

> 把这份文档整理成一份**交付版（fair copy）**。对方是第一次看到它，不需要了解我的修改过程。请：
> 1. 只保留最终内容本身，删掉所有过程痕迹：修订历史 / changelog、版本号与版本说明、`(新增)(已修改)(待确认)` 这类行内批注、给我自己看的 TODO 和备注、以及任何"这版相比上一版改了……"的过程性说明。
> 2. 如果某处留了多个备选写法，每处只保留你判断为最终的那一个，其余删掉。
> 3. 全文按"读者第一次阅读"的视角通顺衔接，不要出现"如前所述的改动""之前的版本里"这种指向历史的措辞。
> 4. **如果有内容你拿不准是过程痕迹还是正文（比如本就属于交付内容的"变更管理"章节），先列出来让我确认，不要直接删。**
> 5. 确认后，直接输出整理好的完整文档，**不要**在开头或结尾解释你删了什么、改了什么。另存为 `xxx-final.md`，原文件别动。

**短版（随手清）：**

> 按"对方第一次看、不关心修改过程"的视角，把这份 md 清成交付版：去掉所有修订历史、版本说明、行内批注和我的 TODO，备选只留最终那版，直接给我干净的完整文档，别解释改了啥，另存别覆盖。

## 例子

**Input（工作稿片段）：**

```markdown
## 3. 技术方案 (v2，已修改)
<!-- TODO: 这段后面要不要拆成两节？先放着 -->
我们采用三层提示词架构。(相比上一版的单层，这版拆成了系统层/章节层/数据层)
~~原来打算用单一长 prompt~~
备选写法A：系统层固化机构规范。
备选写法B：系统层负责角色与全局约束。← 记得问下龚总倾向哪个

### 变更记录
- v1: 初稿
- v2: 应客户要求改为三层架构
```

**Output（fair copy）：**

```markdown
## 3. 技术方案

我们采用三层提示词架构：系统层负责角色与全局约束，章节层定义各章节逻辑，数据层注入结构化输入。
```

（去掉了版本标注、HTML 注释、删除线残留、过程性说明、变更记录章节；备选两版只留了最终判断那一版；"相比上一版"这类指向历史的措辞也清掉。若"变更记录"其实是交付要求的一部分，则应先问而非直接删。）

## 命名与边界（备忘）

- **fair copy（誊清稿）**：草稿经最终修订后誊清、供正式提交的干净版本，与 foul papers（草稿）相对。本 skill 就是把这一步自动化。
- **不要和"去 AI 味"混。** 中文社区搜"去痕迹"大多指降低 AIGC 检测率、让文本不像机器写的——那是改写文风。本 skill 跟文本像不像 AI 无关，只去"改稿过程"留下的元信息。命名上也因此避开"去痕迹"这种说法。
- **跟 `read-digest` 的分工**：`read-digest` 是 **input 侧**——把别人/AI 丢来的臃肿稿读懂、改顺；`fair-copy` 是 **output 侧**——把自己反复改过、混进过程痕迹的稿清成交付版。方向相反。

## 一句话总结

**按"对方第一次看、不关心你怎么改的"视角誊清成品；拿不准的先问再删，工作稿务必留底。**
