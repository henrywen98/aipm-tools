# Fair Copy — 工作稿清成交付稿

> **触发方式：手动。** 这个 skill 会删改正文、另出成品，所以只在用户显式调用 `/fair-copy` 或明确要求"清成交付版"时才动手，不自动介入。

## 这个 skill 在解决什么

一份反复迭代的工作稿，对交付对象来说混着两类噪音：

- **过程信息**：修订历史、改动说明、待确认批注、备选写法、给自己的 TODO。
- **认知负担**：对这个读者无谓的行话、没落地却写成结论的内容、抽象词没说清指谁、前后术语打架。

读者只要"主题本身"。但作者自己往往看不见这两类噪音——这是写作里的 **curse of knowledge（知识的诅咒）**：你 un-know 不掉自己脑子里的上下文，于是留下只有自己懂的指向（"如前所述的改动""上一版里"）、只有内行才懂的行话（工程的"fail-closed"、法律的"对价"、医学的"BID"、学术的"消解了主体性"）、只有你知道还没做的东西（写成了既成事实）。读者一头雾水，但你看不到。

这个 skill 做的事：站到"目标读者第一次冷读"的位置，把这两类噪音都清掉，誊出一份对方一看就顺的成品。适用于任何文体、任何语言——判准和动作是一样的，只是"读者是谁、行话是哪种"随文体变。

**类比锚点**（帮快速理解在干嘛）：
- 手稿学：foul papers（草稿）→ **fair copy（誊清稿）**，定稿后誊清供正式提交。
- 编辑出版：working copy → **clean copy**，编辑常给"带修订追踪版"和"干净版"两份。
- 演讲：写满"这里停顿""看观众""这段再想想"的**彩排讲稿** → 正式发表的讲稿 / 文字稿。
- 开发：一堆 `wip / fix typo / 再改一版` 的乱 commit → **git squash** 成一个连贯干净的 commit。

## 统一判准：冷读测试

逐处只问一个问题：**目标读者第一次看到这里，会不会停下来？**

- 会卡 → 处理它。**减法默认**：能删就删。删不掉（信息是读者要的），就换成他读得懂的说法、或补一句说清楚——**最小改动，绝不为了"润色"而加戏**。
- 不卡 → 别动。

两个一开始就要定的前提：

1. **先定"交付给谁"。** 冷读测试是**相对这个读者**的。同一个词，对内行是精确、对外行是黑话："对价"给律师看正是准词、给客户看是障碍；"fail-closed"给工程读者也许该留、给销售看就是噪音；"BID"给医生留、给家属换成"每天两次"。**按真实读者判断，别一刀切地往下夯到最低水位**——那会把给内行的稿子削成幼儿园读物。
2. **拿不准先问，别静默改。** 尤其是"这是不是没做到 / 做不到""这个术语能不能删"——你未必知道实情。宁可列出来多问一句，不要误伤。

## 要清的两类东西

### 轴一 · 过程痕迹

逐条对照，命中就清（这一轴与文体无关，任何稿子都一样）：

1. 修订历史 / changelog / "更新记录" / "版本演进"段落。
2. 版本号、版本说明（`v0.3`、"本版变更点"、"相比 v0.2 的改进"）。
3. 行内批注：`(新增)``(已修改)``(待确认)``(TODO)``(待补)``(?)``【待核】` 这类。
4. 给自己看的旁批 / 备注 / 提醒（"记得问 XX""这里数据待核""先这么写"）。
5. 过程性指向措辞（diff-anchored writing，"对着 diff 写"）：把内容写成在**叙述一次改动**，而不是**描述事物本身**——"如前所述的改动""上一版里""这版相比上一版…""我后来把它改成了…"。除非文档本身就是版本范畴的（changelog / 发版说明 / 迁移指南），否则正文不该需要"知道上次改了什么"才读得通。
6. 多处并存的备选写法 / "方案 A or 方案 B" / "这句还是那句？"——**每处只保留你判断为最终的那一个**，其余删掉。
7. 注释掉但没删的段落、`<!-- ... -->` HTML 注释、删除线残留（`~~旧文~~`）、高亮标记。

### 轴二 · 冷读障碍 / 认知负担

这一轴清的不是"改稿痕迹"，而是"内容本身让冷读者卡住的地方"。四个维度：

1. **去行话（不限工程）。** 行话 = 对**该读者**的圈内话。工程术语只是其中一类——法律拉丁文与引条、医学缩写、学术黑话、金融行话、政府公文腔、咨询圈套话、公司内部黑话，都算。对该读者不付费的，换白话；术语本身就精确、读者也懂的，留着别动；换不掉又必须用的，第一次出现时一句话点破。
   - 工程 → 业务：`读取层可见性` → `按权限控制谁能看`。
   - 法律 → 外行：`本协议项下之对价` → `这笔报酬`（给律师看则"对价"留着）。
   - 学术 → 大众：`该文本消解了叙事的主体性` → 说清到底是什么意思。
   - 咨询套话 → 实话：`赋能全链路数字化提效` → 具体说帮谁省了什么、省了多少。

2. **求真过滤。** 交付稿里别拿"没落地 / 不现实"的东西当结论。三分处理：
   - **不存在 / 没做的** → 删（别把还没建的能力写成既成事实：提案里"已对接 XX 系统"其实还在试点；简历里"主导"其实只是参与）。
   - **做不到的** → 改成现实做法（别承诺办不到的事："保证 0 误差 / 0 延期"改成能兑现的说法；"随时可回滚"其实不能就说清真实边界）。
   - **真·待办** → 保留，但**诚实标注**（如 `（开发中）``（待第三方确认）`），不藏不夸。

3. **概念分层显性化。** 一个抽象统称对应多个具体身份 / 实例时，把"统称 → 具体"的从属关系当场标清，别让读者卡在"它到底指谁"。
   - 例：一份方案里反复出现"负责人"，实际分指甲方项目经理、我方交付负责人、第三方监理——第一次出现时一句话标清各指谁，别让读者猜。

4. **一致性收口。** 同一份 / 同一组文档里，同一个东西用同一个词。改了一处，**搜一遍全文（及配套件）一起改**，别留半中半英、半新半旧、两图口径打架。
   - 例：同一群人一会儿"用户"一会儿"客户"一会儿"会员" → 统一一个词；两张图里的"现状"口径统一。

## ⚠️ 最容易翻车的地方

误伤比留噪音更严重——丢的是实质内容 / 准确度。逐条防：

- **轴一**：真有一章叫"变更管理 / 修订记录 / 发版说明"且**本身就是交付内容**（合同修订条款、产品文档里读者要看的 changelog、报告要求的版本沿革、论文的修订说明信 response letter）——保留。业务上本就在讲"历史 / 演进 / 迭代"的正文（"该公司近三年战略演进"）——保留。判准：这段是"作者对改稿过程的元说明"还是"读者要的主题内容"？前者删，后者留。
- **去行话**：别把对**真实读者**精确的术语也削平。给工程读者的 `fail-closed`、给医生的 `BID`、给律师的"对价"、给金融读者的 `MNPI`、给学术读者的领域概念，可能正是最准的词。按真实读者判断，拿不准问。
- **求真**：别擅自判定某事"做不到 / 没做"就删——你未必知道实情。列出来问，别替作者下结论。
- **概念分层 / 一致性**：分层是**一句话标清**，不是加一整段；为了"说清"把交付稿写啰嗦，又是另一种噪音。减法仍是默认。

## 工作流

1. **先定交付对象，再判篇幅 / 重要性。** 冷读测试相对读者；读者定了再动手。
   - 短、低风险：直接清。
   - 长 / 重要 / 正式交付：**先列出"我识别到的两类待清项（过程痕迹 + 冷读障碍）+ 求真存疑项"，请用户确认再动手**。这一步专门挡误删。
2. **清理：减法默认。** 能删则删；删不掉的，换白话 / 标分层 / 统一口径——最小改动。严格保留正文实质。
3. **按"读者第一次阅读"重新衔接。** 删掉某段后别留指向它的孤儿句（"如上所述"指向了空气）；该补的衔接补上。
4. **输出语言跟随原稿。** 中文稿出中文、英文稿出英文、中英混排保持原样，不擅自翻译（除非用户要求）；冷读测试也按原稿语言下的目标读者来判。
5. **不要覆盖工作稿。** 输出成新文件（如 `xxx-final.md`），原始工作稿（foul papers）留底——你的思考过程都在里面，下次回溯还要用。
6. **输出干净的完整文档，不在开头 / 结尾解释"我删了啥、改了啥"**——那本身又是一种过程痕迹，交付对象不需要。

## 核心提示词模板

skill 的核心动作，也可直接复制给任何 AI 用。

**完整版（长 / 重要文档）：**

> 把这份文档整理成一份**交付版（fair copy）**。交付对象是【填：客户 / 领导 / 同事 / 公开读者 / 审稿人…】，第一次看到它，不了解我的修改过程、也不一定懂我这一行的术语。判准是"他第一次冷读会不会卡"。请：
> 1. 删掉所有**过程痕迹**：修订历史 / changelog、版本号与版本说明、`(新增)(待确认)` 这类批注、给我自己看的 TODO 和备注、"这版相比上一版改了……"的过程性说明；多个备选写法每处只留最终一版。
> 2. 清掉**冷读障碍**：对这个读者无谓的行话 / 术语换成白话（必须保留的术语第一次出现时点破一句）；没做到 / 不现实的内容别当结论——没做的删、做不到的改成现实做法、真待办保留并标注；抽象统称对应多个具体身份时把关系标清；全文术语口径统一。
> 3. 全文按"读者第一次阅读"通顺衔接，不要出现"如前所述的改动""之前的版本里"这种指向历史的措辞，也不要留指向已删内容的孤儿句。保持原稿语言，别擅自翻译。
> 4. **凡是你拿不准的——是过程痕迹还是正文、某术语能不能删、某事是不是真没做到——先列出来让我确认，不要直接改。**
> 5. 确认后直接输出整理好的完整文档，**不要**在开头或结尾解释你删 / 改了什么。另存为 `xxx-final.md`，原文件别动。

**短版（随手清）：**

> 按"对方第一次冷读、不关心我怎么改的、不一定懂我这行术语"的视角，把这份稿清成交付版：去掉所有修订历史 / 版本说明 / 行内批注 / 我的 TODO（备选只留最终版），行话换白话，没做到 / 不现实的别当结论，术语统一，保持原稿语言。拿不准的先问别擅删。直接给我干净的完整文档，别解释改了啥，另存别覆盖。

**English version（for English or mixed-language drafts）：**

> Turn this document into a clean **fair copy** for a reader who will see it for the first time — audience: 【client / manager / reviewer / public reader…】. They don't know how I revised it and won't necessarily know my field's jargon. The test is: would they stumble on any word or sentence on a cold read? Please:
> 1. Delete all **process residue**: revision history / changelog, version numbers and "what changed" notes, inline flags like `(new)(TBD)(TODO)`, notes-to-self, "compared to the last version…" commentary; where alternate phrasings coexist, keep only the final one.
> 2. Remove **cold-read friction**: replace jargon that doesn't earn its place for this reader with plain language (gloss any term you must keep, on first use); don't state unfinished or unrealistic things as fact — cut what wasn't done, soften what can't be done into what's actually true, keep genuine to-dos but label them honestly; when one umbrella term covers several distinct things, spell out which is which; make terminology consistent throughout.
> 3. Make it read cleanly on first pass — no "as revised above" / "in the previous version" pointers, no orphan sentences referring to deleted content. Keep the original language; don't translate unless asked.
> 4. **Anything you're unsure about — process residue vs. real content, whether a term can go, whether something was actually done — list it and ask me first; don't just change it.**
> 5. Then output the full clean document, with **no** preamble or footnote explaining what you cut or changed. Save as `xxx-final.md`; leave the original untouched.

## 例子

**例 1 · 清过程痕迹（求职简历 / 个人稿）**

Input（工作稿片段）：

```markdown
## 项目经历 (v3，把带货那段挪上来了)
<!-- 这段是不是太长？投大厂前再删 -->
主导（其实是参与，措辞待定）某电商 App 改版，上线后 DAU +20%（这个数要跟老王核一下）。
备选：· 负责核心流程重构   · 主导核心流程重构 ← 用哪个
~~2019 便利店兼职~~（太旧先删）
```

Output（fair copy）：

```markdown
## 项目经历
参与某电商 App 改版，负责核心流程重构。
```

（**过程痕迹**全清：版本标注、HTML 注释、删除线残留、备选写法、"要跟老王核一下"旁批、"把带货那段挪上来"改动说明；**求真**："主导"其实是参与 → 按事实定"参与"，"DAU +20%（待核）"没核实别写成结论——核实了再放具体数，没核就先不写。）

**例 2 · 去行话（法律条款 → 给非法律读者的说明版）**

Input（工作稿片段）：

```markdown
如乙方未能于本协议项下约定期限内履行给付义务，甲方有权主张违约金，
数额为合同对价之 20%，且不影响其就实际损失（含但不限于可得利益损失）另行主张之权利。
```

Output（fair copy）：

```markdown
如果对方没按约定时间付款，你可以要一笔违约金，金额是合同总额的 20%。
这笔违约金之外，你实际的损失（包括本来能赚到、却因此没赚到的部分）还可以另外索赔。
```

（"本协议项下之给付义务"→"按约定付款"、"对价"→"合同总额"、"可得利益损失"→"本来能赚到却没赚到的"。注意：若交付给**律师同行、或作为正式签署文本**，这些术语正是精确措辞，该留——冷读测试永远相对真实读者，不是越白越好。）

**例 3 · 求真 + 去行话（商业提案 → 交付给客户 / 老板）**

Input（工作稿片段）：

```markdown
## 4. 交付能力（待老板过目）
我们已赋能客户全链路数字化提效，对接了 ERP、CRM、BI 三大系统（BI 其实还在 POC）。
交付负责人对所有节点 fail-closed 把关，保证 0 延期。
※ 这段是不是写太满了？先放着 —— 记得跟报价口径对一下
```

Output（fair copy）：

```markdown
## 4. 交付能力
我们已对接客户的 ERP 与 CRM 系统，并在推进 BI 系统的试点接入。
项目由专职交付负责人统一把关各节点进度。
```

（**过程痕迹** "待老板过目"、`※` 旁批、"记得跟报价口径对一下"全清；**去行话** "赋能全链路数字化提效"→说清做了啥，"fail-closed"对业务读者删；**求真** BI 还在 POC→改成"推进试点"而非写成已完成，"保证 0 延期"办不到→删。）

**例 4 · 英文稿 / 中英混排（跨语言）**

Input（工作稿片段）：

```markdown
## Results (v3 — reran after fixing the seed bug)
Our method beats the baseline by a wide margin (see Table 2; earlier draft said +2.1, now +3.4 after the fix).
TODO: double-check the p-value with the stats team before submission
we're basically SOTA now lol
```

Output（fair copy）：

```markdown
## Results
Our method outperforms the baseline by 3.4 points (Table 2).
```

（删版本标注与"reran after fixing…"过程说明、"earlier draft said +2.1"的改动叙述、`TODO` 旁批；"we're basically SOTA now lol"是口语过程话，删，只留可交付的客观陈述。**英文稿出英文，不译成中文。**）

## 命名与边界（备忘）

- **fair copy（誊清稿）**：草稿经最终修订后誊清、供正式提交的干净版本，与 foul papers（草稿）相对。本 skill 把这一步自动化，并把判准从"去痕迹"扩成"过冷读测试"。
- **去行话 ≠ 去 AI 味。** 去行话是"让**这个读者**读得懂术语"，跟文本像不像 AI 无关；中文社区说的"去痕迹 / 去 AI 味"大多指降低 AIGC 检测率、让文本不像机器写——那是改写文风，归 `humanizer`。**分工**：先 `fair-copy` 清噪音 + 落白话，再按需 `humanizer` 去 AI 味。命名上也因此避开"去痕迹"这种含混说法。
- **跟 `read-digest` 的分工**：`read-digest` 是 **input 侧**——把别人 / AI 丢来的臃肿稿读懂、改顺；`fair-copy` 是 **output 侧**——把自己反复改过、混进过程痕迹与认知负担的稿，清成对方冷读就顺的交付版。方向相反。

## 一句话总结

**按"目标读者第一次冷读会不会卡"誊清成品——任何文体、任何语言都一样：卡住的要么删、要么换白话、要么说清楚；没做到 / 不现实的别当结论；减法默认，拿不准先问，工作稿务必留底。**
