# fair-copy

> 把工作稿誊清成交付稿——目标读者第一次冷读就顺。任何文体、任何语言。

你自己反复改过很多版的稿子，对交付对象来说混着两类噪音：**过程痕迹**（修订历史、版本说明、行内批注、TODO、并存的备选写法）和**认知负担**（对这个读者无谓的行话、没落地却写成结论的内容、抽象词没说清指谁、前后术语打架）。`fair-copy` 站到「读者第一次冷读」的位置，把这两类都清掉，誊出一份对方一看就顺的成品（fair copy / 誊清稿）。适用于工程 / PM 文档、学术报告、法律与商业文书、营销文案、个人 / 求职材料，中文、英文或中英混排都行。

## 为什么需要它

作者自己往往**看不见**这些噪音——这就是写作里的 *curse of knowledge（知识的诅咒）*：你 un-know 不掉自己脑子里的上下文，于是留下只有自己懂的指向（"如前所述的改动"）、只有内行懂的行话（工程的"fail-closed"、法律的"对价"、医学的"BID"、学术的"消解了主体性"）、只有你知道还没做的东西（写成了既成事实）。`fair-copy` 替你切换到读者视角，做这一步誊清。

类比：手稿学的 foul papers → **fair copy**；编辑出版的 working copy → **clean copy**；写满"这里停顿""看观众"的彩排讲稿 → 正式讲稿；开发里一堆乱 commit → **git squash** 成一个干净 commit。

## 统一判准：冷读测试

逐处只问一句：**目标读者第一次看到这里，会不会停下来？**

- 会卡 → 处理它。**减法默认**：能删就删；删不掉就换白话 / 说清楚，最小改动、不为润色加戏。
- 不卡 → 别动。

两个前提：① **先定交付给谁**——冷读测试相对这个读者，"对价"给律师是准词、给客户是障碍，"fail-closed"对工程读者也许该留、对销售是噪音，别一刀切往下夯到最低水位；② **拿不准先问**，尤其"是不是没做到 / 能不能删某术语"，别替作者下结论。

## 它清掉什么

**轴一 · 过程痕迹**
- 修订历史 / changelog / "更新记录" / "版本演进"
- 版本号、版本说明（`v0.3`、"相比 v0.2 的改进"）
- 行内批注：`(新增)(已修改)(待确认)(TODO)(待补)【待核】`
- 给自己看的旁批 / 备注 / 提醒
- "这版相比上一版改了……"这类过程性指向措辞
- 多处并存的备选写法（每处只留最终那一版）
- `<!-- HTML 注释 -->`、删除线残留 `~~旧文~~`、高亮标记

**轴二 · 冷读障碍 / 认知负担**
- **去行话（不限工程）**：对该读者无谓的术语 / 行话换白话——工程术语只是其一，法律、医学、学术、金融、政府公文、咨询套话、公司内部黑话同理（必须保留的第一次出现时点破一句）。例：`读取层可见性`→`按权限控制谁能看`、`本协议项下之对价`→`这笔报酬`（给律师则留）、`赋能全链路提效`→说清帮谁省了什么。
- **求真过滤**：没做的删、做不到的改成现实做法、真·待办保留并诚实标注。例：提案里"已对接 XX 系统"其实还在试点 → 改成"推进试点"；"保证 0 延期"办不到 → 删；`XX 模块（开发中）`真待办 → 保留标注。
- **概念分层**：抽象统称对应多个具体身份时，把"统称→具体"标清。例：一份方案里反复出现的"负责人" → 甲方项目经理 / 我方交付负责人 / 第三方监理，一句话标清各指谁。
- **一致性收口**：同一份 / 同组文档术语口径统一，改一处搜全文一起改。例：同一群人别一会儿"用户"一会儿"客户"一会儿"会员"。

## 严格保留（别误删）

误伤比留噪音更严重。真属于交付内容的"变更管理 / 修订条款 / 产品 changelog / 论文 response letter"章节、业务在讲"历史 / 演进"的正文——保留。对**真实读者**精确的术语（给工程的 `fail-closed`、给医生的 `BID`、给律师的"对价"、给金融的 `MNPI`）——保留。拿不准的某事是否"做不到 / 没做"——先问，别擅删。

## 触发方式（手动）

`fair-copy` 会删改正文、另出成品，所以**只手动触发、不自动介入**——避免在你没要求时擅自动你的稿。用以下任意一种明确唤起（中英文都行）：

- 直接 `/fair-copy`
- "把这份稿清成交付版 / 誊清版 / 干净成品" / "整理成给客户 / 领导 / 审稿人看的版本"
- "去掉我的修改记录 / 修订历史 / 版本说明，做成干净成品稿"

单纯说"改一下 / 润色一下"不触发——它要的是"清成给别人看的交付版"这个明确意图。

## 工作流程

```
1  先定交付对象，再判篇幅 / 重要性
   - 短 / 低风险 → 直接清
   - 长 / 重要 / 正式交付 → 先列"两类待清项 + 求真存疑项"请你确认再动手
2  减法默认：能删则删；删不掉的换白话 / 标分层 / 统一口径，严格保留正文
3  按"读者第一次阅读"重新衔接全文（不留指向已删段落的孤儿句）
4  输出语言跟随原稿——中文出中文、英文出英文，不擅自翻译
5  输出到新文件 xxx-final.md，原工作稿留底（foul papers 不覆盖）
6  直接给干净的完整文档，不在开头 / 结尾解释"删了啥、改了啥"
```

## 边界（重要区分）

- **去行话 ≠ 去 AI 味。** 去行话是"让这个读者读得懂术语"，跟文本像不像 AI 无关；降 AIGC 检测率、让文本不像机器写是改写文风，归 `humanizer`。分工：先 `fair-copy` 清噪音 + 落白话，再按需 `humanizer` 去 AI 味。
- **跟 `read-digest` 的分工**：`fair-copy` 是 **output 侧**——清自己反复改过的稿；`read-digest` 是 **input 侧**——读懂 / 改顺别人或 AI 丢来的臃肿稿。方向相反，互补成对。

## 一句话总结

**按"目标读者第一次冷读会不会卡"誊清成品——任何文体、任何语言都一样：卡住的要么删、要么换白话、要么说清楚；没做到 / 不现实的别当结论；减法默认，拿不准先问，工作稿务必留底。**
