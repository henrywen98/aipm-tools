# fair-copy

> 把工作稿誊清成交付稿——读者第一次看就是成品。

一个**只负责"清痕迹"**的 skill：你自己用 AI 反复改了很多版的稿子，里面混着修订历史、版本说明、行内批注、给自己看的 TODO、并存的备选写法……这些是**给你自己看的过程信息**，交付对象不需要。`fair-copy` 按「读者第一次冷读」的视角，把这些过程痕迹剥干净，誊出一份干净成品（fair copy / 誊清稿）。

## 为什么需要它

作者自己往往**看不见**这些痕迹的存在——这就是写作里的 *curse of knowledge（知识的诅咒）*：你没法 un-know 自己脑子里的上下文，于是不自觉留下只有自己懂的指向（"如前所述的改动""上一版里"），读者一头雾水。`fair-copy` 替你切换到读者视角，做这一步誊清。

类比：手稿学的 foul papers → **fair copy**；编辑出版的 working copy → **clean copy**；编剧的 starred draft → **clean draft**；开发里一堆乱 commit → **git squash** 成一个干净 commit。

## 它清掉什么 / 保留什么

**清掉（过程痕迹）：**
- 修订历史 / changelog / "更新记录" / "版本演进"
- 版本号、版本说明（`v0.3`、"相比 v0.2 的改进"）
- 行内批注：`(新增)(已修改)(待确认)(TODO)(待补)【待核】`
- 给自己看的旁批 / 备注 / 提醒
- "这版相比上一版改了……"这类过程性指向措辞
- 多处并存的备选写法（每处只留最终那一版）
- `<!-- HTML 注释 -->`、删除线残留 `~~旧文~~`、高亮标记

**严格保留（正文）：**
- 真正属于交付内容的"变更管理 / 修订记录 / 发版说明"章节（合同修订条款、产品 changelog 等）
- 业务上本就在讨论"历史 / 演进 / 迭代"的正文
- **判断标准**：是"作者对自己改稿过程的元说明"（删），还是"读者需要的、关于主题本身的内容"（留）。**拿不准就先问，不静默删。**

## 触发方式

直接说出以下任意一种（中英文都行）：

- "清成交付版" / "整理成给客户/领导看的版本"
- "去掉我的修改记录 / 修订历史 / 版本说明"
- "这个 md 给别人看之前清一下"
- "别人不需要知道我中途怎么改的，只要一个最终版"
- "把过程痕迹去掉" / "做成干净的成品稿"
- `fair copy` / `clean copy`

或者直接 `/fair-copy`。

## 工作流程

```
1  判断篇幅与重要性
   - 短 / 低风险 → 直接清
   - 长 / 重要 / 正式交付 → 先列出"识别到的所有过程痕迹清单"请你确认再动手
2  只去过程痕迹，严格保留正文实质内容
3  按"读者第一次阅读"重新衔接全文（不留指向已删段落的孤儿句）
4  输出到新文件 xxx-final.md，原工作稿留底（foul papers 不覆盖）
5  直接给干净的完整文档，不在开头/结尾解释"删了啥、改了啥"
```

## 边界（重要区分）

- **不要和"去 AI 味"混。** 中文社区搜"去痕迹"大多指降低 AIGC 检测率、让文本不像机器写的——那是改写文风。`fair-copy` 跟文本像不像 AI 无关，只去"改稿过程"留下的元信息。
- **跟 `read-digest` 的分工**：`fair-copy` 是 **output 侧**——清自己反复改过的稿；`read-digest` 是 **input 侧**——读懂 / 改顺别人或 AI 丢来的臃肿稿。方向相反，互补成对。

## 一句话总结

**按"对方第一次看、不关心你怎么改的"视角誊清成品；拿不准的先问再删，工作稿务必留底。**
