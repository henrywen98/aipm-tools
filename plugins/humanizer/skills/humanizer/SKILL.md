# Humanizer — 让 AI 写的文字变成人能读懂的

## 根问题：AI 写的文字，不像一个真人在为读者写

大模型靠"猜下一个最可能的词"生成文本，天然堆出一套机器腔：拔高升华、三段排比、黑话、名词化的长句、成串被动、套话开场和空洞收尾。翻成中文时更僵——名词化、从句嵌套、一句三个"的"，读着像英译中的说明书。加上作者自己看不见的**知识的诅咒**（curse of knowledge）：留下只有内行懂的行话、只有自己知道还没做的事、只有自己看得懂的旧版指向。结果就是读者一读就卡，但作者看不到。

这个 skill 做一件事：**站到"目标读者第一次冷读"的位置，把这些卡点清掉，交出一份对方一看就顺的稿。** 中文、英文、中英混排都一样，判准和动作不变，只是"读者是谁、行话是哪种"随稿子变。

## 判准：冷读测试

逐处只问一句：**目标读者第一次看到这里，会不会停下来？**

- 会卡 → 处理它。**减法默认**：能删就删。删不掉（信息是读者要的），就换成他读得懂的说法、或补一句说清楚——最小改动，不为"润色"加戏。
- 不卡 → 别动。

两个前提先定：

1. **先定"交付给谁"。** 冷读测试是**相对这个读者**的。同一个词，对内行是精确、对外行是黑话："对价"给律师是准词、给客户是障碍；`fail-closed` 给工程读者也许该留、给销售就是噪音。**按真实读者判断，别一刀切往下夯到最低水位**，那会把给内行的稿削成幼儿园读物。
2. **拿不准先问，别静默改。** 尤其"这是不是没做到 / 做不到""这个术语能不能删"——你未必知道实情。宁可多问一句，别误伤。

## 三层要清的东西

### 层一 · 机器发声（去 AI 味 + 去翻译腔）

句子本身"不像人写的"。两类一起抓，**完整反例 → 正例见 `references/patterns-zh.md`（中文）/ `references/patterns-en.md`（英文）**。

**A. AI 味（靠 cluster 判断，见下方误判防护）**
- 拔高升华：标志着……的重要里程碑、为……奠定坚实基础、开启新篇章
- 三段排比 / 否定式排比：不仅……而且、不是……而是、"创新、协作、共赢"
- 互联网黑话：赋能、抓手、闭环、纵深、重塑、链路、心智、颗粒度、对齐、拉通、沉淀（扎堆即中招）
- 全角破折号"——"成串（基本清掉，换句号 / 逗号 / 冒号 / 括号）
- 模糊归因（"业内人士认为"）、机械加粗成片、emoji 装饰、inline-header 列表
- 空洞收尾：总而言之、前景广阔、未来可期、必将迈向……

**B. 翻译腔 / 中文不顺（对照 Funda 的中文写作规矩，命中就改）**
- **名词化改回动词**："进行了优化"→"优化了"；"对 X 进行处理"→"处理 X"；"实现了……的功能"→"能……"。
- **长句拆短**，一句只说一件事；少用"在……的情况下，由于……，因此……"这种从句套从句。
- **少被动**："X 被 Y 处理"→"Y 处理 X"；成串无主语 + "得到有效落实"的公文腔，点明主体改主动。
- **一句里别连着三个"的"**；能用"和"就别用"以及"；"首先 / 其次 / 最后"除非真在列步骤，否则删。
- **空修饰删掉**：非常、极其、相当、所谓的、相关的、对应的——大多能删。
- **禁用套话词**（直接删或换白话）：值得注意的是、需要指出的是、总的来说、简而言之、基本上、事实上、换句话说；客服腔"希望对你有帮助""如需……请告诉我"；谄媚"这是个好问题""你说得非常对"。

### 层二 · 读者读不懂（冷读障碍）

句子像人写的了，读者还是可能卡——卡在"看不懂"上。

1. **去行话，换白话。** 行话 = 对**该读者**的圈内话，不限工程：法律拉丁文、医学缩写、学术黑话、金融行话、公司内部黑话都算。对该读者不付费的换白话；术语本身精确、读者也懂的留着；换不掉又必须用的，第一次出现一句话点破。
   - 工程 → 业务：`读取层可见性` → `按权限控制谁能看`。
   - 三条硬规则（历史误伤最重，命中就执行）：
     - **读者语言是中文** → 英文缩写与术语换成中文说法，或首次出现加一句中文注释。`BFF`、`fail-closed` 对中文读者全是卡点，"接手人是开发多半懂"不成立——按读者的**语言**判，别按职业猜。
     - **文中给事物起了名 / 分了类**（表头、图例、状态名）→ 用**读者自己现有的叫法和分类法**，不用代码或作者视角的叫法；拿不准就列出来问。
     - **正文引用内部编号 / 章节号**（`§5.6`、`F9.3`、`R2`）→ 每处当场一句话自解释（"F9.3（会邀费率显示的那个 bug）"），或换成描述本身。读者手上没有你的编号索引。
   - 错别字（`瑮华`→`集华`）是最基本的卡点，直接改，改了哪些在拍板项里一并列出。
2. **说清指代（概念分层）。** 一个抽象统称对应多个具体身份时，第一次出现就一句话标清各指谁。例：一份方案里反复出现"负责人"，实际分指甲方项目经理、我方交付负责人、第三方监理——别让读者猜。
3. **术语前后统一。** 同一份 / 同一组文档里，同一个东西用同一个词。改了一处，**搜一遍全文（及配套件）一起改**，别留半中半英、半新半旧、两图口径打架（"用户 / 客户 / 会员"混用 → 统一一个）。
4. **结构冷读（按文档类型套默认结构）。** 卡点也在信息摆放的顺序：
   - **PRD / spec**：业务内容在上；决策记录、技术锚点、代码对应沉到底部（或子页）；User Story 与验收标准每条压成 1–2 句。
   - **表格 / 矩阵**：行列名用读者的分类法；每格一句话收口；读者读不出结论就按"他要回答的问题"重排行列，别按数据来源排。
   - **交接 / 说明类**：结论与"读者第一步要干什么"前置，背景细节在后。
   - 结构重排**有限度**：只动摆放顺序和层级，不改内容实质；整篇推倒重写是禁区（见误判防护）。

### 层三 · 过程噪音（仅"自己反复改过的稿"才有）

这层跟 AI 无关，是自己迭代留下的痕迹。命中就清（任何稿子都一样）：

1. 修订历史 / changelog / "更新记录" / "版本演进"段落。
2. 版本号、版本说明（`v0.3`、"相比 v0.2 的改进"、"本版变更点"）。
3. 行内批注：`(新增)(已修改)(待确认)(TODO)(待补)【待核】(?)`。
4. 给自己看的旁批 / 备注 / 提醒（"记得问 XX""这里数据待核""先这么写"）。
5. **对着 diff 写**的过程性指向措辞："如前所述的改动""上一版里""这版相比上一版……""我后来改成了……"。除非文档本身就是版本范畴的（changelog / 发版说明 / 迁移指南），正文不该要"知道上次改了什么"才读得通。
6. 多处并存的备选写法 / "方案 A or B" / "这句还是那句？"——**每处只留你判断为最终的那一个**，其余删掉。
7. 注释掉没删的段落、`<!-- ... -->`、删除线残留（`~~旧文~~`）、高亮标记。

**求真过滤**（贯穿三层）：交付稿别拿"没落地 / 不现实"的东西当结论。三分处理——**没做的**删（别把还没建的写成既成事实）；**做不到的**改成现实做法（"保证 0 延期"办不到 → 删或改成能兑现的说法）；**真·待办**保留但诚实标注（`（开发中）``（待第三方确认）`）。

## ⚠️ 误判防护（别把好稿改坏）

误伤比留噪音更严重——丢的是实质内容和准确度。逐条防：

- **AI 味看 cluster，不看单点。** 正常人也会命中某一条。文笔干净、偶尔一个"然而"、偶尔一个全角破折号、没有出处——单独出现都**不是**证据。全角破折号 + 三段排比 + "赋能闭环" + "展望未来"结尾一起出现，才是确凿的 AI 味。
- **别削平对真实读者精确的术语。** 给工程读者的 `fail-closed`、给医生的 `BID`、给律师的"对价"、给金融读者的 `MNPI`，可能正是最准的词。按真实读者判断，拿不准问。
- **求真别擅自下结论。** 别自己判定某事"做不到 / 没做"就删——你未必知道实情，列出来问。
- **说清指代 / 统一是"一句话"，不是加一段。** 为了"说清"把稿写啰嗦，又是另一种噪音。减法仍是默认。
- **长文档只做定点编辑，别整篇重写。** "重新组织语言全文重写"必然引入信息错乱和事实漂移，作者会对产出整体失去信任。找到命中项逐处小改，其余一字不动。结构冷读也只动摆放顺序和层级。
- **场景反转的保留清单。** 交接、审计、合规类文档里，日期 / 署名 / changelog 可能正是读者要的交付内容（与层三默认全删相反）——这类先确认保留清单再动手，归入拍板项。真有一章叫"变更管理 / 修订记录"且**本身就是交付内容**（合同修订条款、产品 changelog、论文 response letter），保留。判准：这段是"作者对改稿过程的元说明"还是"读者要的主题内容"？前者删，后者留。

## 别改成"没灵魂的干净稿"

去掉机器味只是一半。**句句一样长、只中立陈述、不带任何观点的稿，跟 AI 味一样一眼假。** 仅在内容和口吻允许时（博客 / 随笔 / 观点 / 个人化文字）注入人味：有观点、句子长短交错、容许一点"乱"（题外话、半成型的想法）。**但公文 / 技术文档 / 法律 / 百科类，中性朴素本身就是正确的人类声音，别硬塞第一人称和观点。**

**人写的信号，倾向保留**（过度改写会改没了灵魂）：具体到难以编造的细节（真实地名 / 怪异引语 / 私人小事）、复杂矛盾的情绪、有年代感的梗、作者能解释"为什么这么写"的选择、真诚的插话和自我更正。

## 工作流：draft → audit → final

0. **先锁定目标稿。** 没带文件 / 链接 → 从上下文推断对象（最近在编辑或讨论的那份），一句话确认（"我按 XX 来清，对吗？"）再动手，别自选一个开审。
1. **定交付对象（读者），判篇幅 / 重要性。** 冷读测试相对读者，读者定了再动。
2. **写 draft，保留原意和覆盖面。** 原文几段、改完还是几段，别越改越短丢信息。机器腔换成具体事实和朴素句子（多用"是 / 有"），匹配原文该有的语域。
3. **audit：自问"这段为什么还不像人写的 / 读者会卡在哪"。** 一两句点出残留的 tell 和卡点。
4. **写 final，把 audit 点出的都改掉。** 中文稿收尾搜一遍 `——`；英文稿搜 `—`、`–`，有残留就是没改完。
5. **两档确认，一次问完。**
   - **默认清理包**：三层的明确命中项（AI 味 / 翻译腔 / 行话替换 / 编号自解释 / 错别字 / 过程痕迹 / 结构默认）打包成"我将直接执行"的一段话，不逐项征询。短、低风险的稿连清单都不列，直接清。
   - **拍板项**：只有求真存疑（"这是不是没做到"）、术语去留拿不准、场景反转（交接稿要不要保留署名 / 日期 / changelog）这类真需要作者判断的，单独列出，通常 ≤3 条。
6. **输出语言跟随原稿。** 中文出中文、英文出英文、中英混排保持原样，不擅自翻译（除非用户要求）。
7. **留底按载体，不再问。**
   - Notion 页 / 在线文档：就地编辑——平台版本历史就是留底。
   - SVG / 图：就地覆盖，只留一版。
   - 本地 md / docx：另存新文件（如 `xxx-final.md`），原始工作稿留底。
8. **交付干净的完整文档，不在开头 / 结尾解释"我删了啥、改了啥"**——那本身又是一种过程痕迹，读者不需要。用户要看时再附 draft 和简短的改动清单。

## 可选：声音校准（voice calibration）

用户给了 2-3 段他自己以前写的东西，就先分析再改：句子长短习惯、用词正式度、怎么起段、标点习惯、口头禅。改写时不只是去机器味，而是**换上样本里的写法**——他写短句就别给长句，他用"东西 / 搞"就别升级成"要素 / 构建"。没给样本就用默认（自然、有变化、敢有观点）。

## 一句话总结

**按"目标读者第一次冷读会不会卡"清成品：三层一起清——机器发声（去 AI 味 + 去翻译腔）、读者读不懂（换白话 / 说清指代 / 术语统一 / 结构摆顺）、过程噪音（修订史 / 批注 / 备选）。AI 味看 cluster、别见词就改；没做到 / 不现实的别当结论；减法默认、定点编辑；明确的直接清、拿不准的一次问完；去味之余别把人味也改没了。**
